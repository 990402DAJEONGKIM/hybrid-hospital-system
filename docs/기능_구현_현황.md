### 3-2. 인증 보안 정책

| 요구사항 | 구현 상태 | 세부 내용 |
|---|---|---|
| 비밀번호 복잡도 검증 | ✅ | 8자 이상, 대/소/숫자/특수 중 2종류 이상 |
| 비밀번호 만료 관리 | ✅ | 90일 만료, `password_expires_at` 트리거 자동 계산 |
| 비밀번호 정책 동적 로드 | ✅ | `password_policy` 테이블에서 실시간 적용 |
| 로그인 실패 계정 잠금 | ✅ | 5회 실패 시 30분 잠금 |
| httpOnly 쿠키 JWT | ✅ | XSS 방어, SameSite=Lax |
| bcrypt 비밀번호 해싱 | ✅ | 평문 저장 없음 |
| API Key 검증 | ✅ | `X-API-Key` 헤더 |
| 로그인 이력 기록 | ✅ | `login_history` 테이블 (성공/실패/잠금) |
| 계정 잠금 시 admin SES 알림 | ❌ | SES 연동 미완료 |
| 세션 만료 5분 전 경고 팝업 | ❌ | 프론트엔드 미구현 |

### 4-2. 예약 유형별 비식별화 규칙 (코드 레벨 강제)

| 예약 유형 | 허용 입력 필드 | 차단 필드 |
|---|---|---|
| 초진 | `department_code`, `appointment_date`, `appointment_time`, `notes` | EMR 전체 |
| 재진 | + `doctor_id` (선택) | 이름, 주민번호, 진단 원문 |
| 입원 | + `room_type_pref`, `has_chronic_condition`(bool) | 기저질환명 원문 |
| 수술 전 | 위와 동일 (`pre_surgery` type_code) | 수술명/알레르기명 원문 |

> 재진·수술전은 `requires_previous_visit=TRUE` — 해당 진료과 내원 이력 없으면 400 반환

## 9. 아키텍처 요약

```
[환자 브라우저]                    [스태프 브라우저]
      │                                   │
      ▼ HTTPS                             ▼ HTTPS (WAF IP 제한)
[Route53] → [ALB]               [Route53] → [ALB]
      │                                   │
      ▼                                   ▼
[ECS Task]                        [ECS Task]
  ├── nginx (정적 서빙 + API 프록시)   ├── nginx
  └── FastAPI (8000)               └── FastAPI (8000)
            │                               │
            └───────────────────────────────┘
                           │
                           
               [Aurora PostgreSQL 17.x]
               (Private Subnet, 5432)
                           ▲
                  pglogical 단방향 복제 (온프레미스 → AWS)
                  AWS → 온프레미스 접근 완전 차단
                           │
               [온프레미스 EMR DB]
               (1등급 PII 원본 보관)
```

**설계 핵심 원칙**
- 1등급 PII (성명·주민번호·전화번호·진단원문)는 온프레미스에만 존재
- AWS에는 비식별화 해시 + 코드값만 복제
- 예약·인증·포털은 AWS, 실제 의료 기록은 온프레미스
