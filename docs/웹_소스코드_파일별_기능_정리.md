## 백엔드 (backend/)

### 루트 파일

| 파일 | 역할 | 주요 기능 |
|------|------|-----------|
| `main.py` | **앱 진입점** | FastAPI 앱을 생성하고 미들웨어를 등록하는 파일. 모든 HTTP 요청은 이 파일을 통해 처리 순서가 결정됨. TrustedHost(허용 도메인 필터) → 감사 로그 → 보안 헤더 → GZip 압축 → CORS 순으로 처리 |
| `requirements.txt` | **패키지 목록** | 서버 실행에 필요한 Python 라이브러리 목록. `pip install -r requirements.txt` 명령 한 번으로 전체 설치 가능 |
| `.env` | **환경변수 (로컬 개발용)** | DB 접속 주소, JWT 서명 키, API 키, 쿠키 보안 설정 등 환경마다 달라지는 민감한 값 모음. **Git에 올리면 안 됨** |
| `.env.example` | **환경변수 템플릿** | `.env` 파일을 어떻게 작성해야 하는지 보여주는 예시. 신규 팀원이 환경 세팅 시 이 파일을 복사해서 `.env`로 사용 |
| `create_admin.py` | **초기 관리자 계정 생성 스크립트** | 서버 최초 배포 시 1회만 실행. `admin@hospital.com` 관리자 계정을 DB에 직접 생성. 이후 의사·간호사 계정은 관리자 페이지에서 생성 |

### core/ (핵심 공통 모듈)

모든 라우터가 공통으로 사용하는 DB 연결, 인증, 미들웨어 로직을 모아둔 폴더.

| 파일 | 역할 | 주요 기능 |
|------|------|-----------|
| `core/database.py` | **DB 연결 관리** | PostgreSQL 연결 엔진을 생성하고 세션을 관리. 각 API 요청마다 DB 연결을 열고 응답이 끝나면 닫는 `get_db()` 함수를 제공. `.env`의 `DATABASE_URL` 값을 읽어 연결 |
| `core/security.py` | **인증·보안 핵심 로직** | ① 비밀번호 bcrypt 해싱 및 검증 ② JWT access token 생성·검증 ③ 랜덤 refresh token 생성 ④ API 키 검증 (`X-API-Key` 헤더) ⑤ httpOnly 쿠키에서 현재 로그인 사용자 추출 (`get_current_user`) |
| `core/middleware.py` | **미들웨어 2종** | **감사 로그 (AuditLogMiddleware)**: 모든 API 호출을 `audit_logs` 테이블에 기록 — 누가 / 언제 / 어떤 요청 / 결과가 무엇인지 (ISMS-P 2.9.1 대응). **보안 헤더 (SecurityHeadersMiddleware)**: HSTS, CSP, X-Frame-Options 등 7가지 보안 헤더를 모든 HTTP 응답에 자동 추가 (ISMS-P 2.10.1 대응) |

### models/ (데이터베이스 테이블 정의)

| 파일 | 역할 | 정의된 테이블 |
|------|------|--------------|
| `models/db.py` | **ORM 모델** | DB 테이블과 Python 클래스를 1:1 매핑. 아래 10개 테이블을 정의 |

**정의된 테이블 목록**

| 클래스명 | 테이블명 | 설명 |
|----------|----------|------|
| `User` | `users` | 로그인 계정 정보 (이메일, 비밀번호 해시, 역할, 잠금 상태 등) |

| `SyncDepartment` | `sync_departments` | 진료과 목록 (내과, 외과 등) |

| `SyncEncounter` | `sync_encounters` | 예약 및 진료 기록 (방문일, 상태코드, 담당 의사) |

| `AuditLog` | `audit_logs` | 모든 API 호출 감사 로그 |

### routers/ (API 엔드포인트)

실제 비즈니스 로직이 구현된 API 파일들. URL prefix 기준으로 파일이 분리되어 있음.

#### `routers/auth.py` — 인증 API (`/auth/`)

| 엔드포인트 | HTTP 메서드 | 기능 | 접근 권한 |
|------------|-------------|------|-----------|
| `/auth/register` | POST | 환자 회원가입. 생년+성별로 EMR 환자 DB 자동 매칭 시도 | 누구나 |
| `/auth/login` | POST | 로그인. 성공 시 JWT access token + refresh token을 **httpOnly 쿠키**로 발급 | 누구나 |
| `/auth/refresh` | POST | 쿠키의 refresh token으로 새 token 발급 (Rotation 방식 — 구 토큰 즉시 폐기) | 누구나 |
| `/auth/logout` | POST | 로그아웃. DB 세션 폐기 + 쿠키 삭제 | 누구나 |
| `/auth/me` | GET | 현재 로그인 사용자 정보 조회 (role, 비밀번호 만료 여부) | 로그인 필요 |
| `/auth/change-password` | POST | 비밀번호 변경. 완료 후 해당 계정의 **모든 세션 강제 폐기** | 로그인 필요 |

> **보안 포인트**: 로그인 실패 5회 누적 시 30분 계정 잠금, 비밀번호 90일마다 만료 강제 (ISMS-P 2.5.3 / 2.5.4)

---

#### `routers/portal.py` — 환자·의료진 포털 API (`/portal/`)

**환자용 API** (role = `patient`만 접근 가능)

| 엔드포인트 | HTTP 메서드 | 기능 |
|------------|-------------|------|
| `/portal/appointments` | GET | 내 예약 목록 전체 조회 |
| `/portal/appointments` | POST | 신규 예약 생성 (진료과, 날짜 선택) |
| `/portal/appointments/{id}` | GET | 예약 단건 상세 조회 |
| `/portal/appointments/{id}` | PATCH | 예약 수정 (OPEN 상태만 가능) |
| `/portal/appointments/{id}` | DELETE | 예약 취소 (OPEN 상태만 가능) |

**의료진용 API** (role = `doctor` / `nurse` / `admin`만 접근 가능)

| 엔드포인트 | HTTP 메서드 | 기능 |
|------------|-------------|------|
| `/portal/doctor/schedule` | GET | 담당 의사의 진료 일정 조회 |
| `/portal/doctor/patients` | GET | 담당 환자 목록 (출생연도, 성별, 마지막 방문일) |
| `/portal/doctor/patients/{hash}` | GET | 환자 상세 정보 (진단 이력, 알레르기, 수술 이력, 과거 진료 기록) |

---

#### `routers/admin.py` — 관리자 API (`/admin/`)

| 엔드포인트 | HTTP 메서드 | 기능 | 접근 권한 |
|------------|-------------|------|-----------|
| `/admin/users` | POST | 의사·간호사·관리자 계정 직접 생성 | `admin` role 전용 |

## 프론트엔드 (frontend/)

서버 없이 브라우저에서 직접 실행되는 HTML/CSS/JS 파일. 백엔드 API를 호출해 데이터를 주고받음.

### HTML 페이지

| 파일 | 역할 | 주요 기능 |
|------|------|-----------|
| `login.html` | **로그인 페이지** | 이메일·비밀번호 입력 → `/auth/login` 호출 → 서버가 httpOnly 쿠키 자동 발급 → 비밀번호 만료 여부 확인 후 이동 |
| `signup.html` | **회원가입 페이지** | 이름·이메일·비밀번호·생년월일·성별 입력 → `/auth/register` 호출. 생년+성별로 기존 환자 DB 매칭 시도 |
| `change-password.html` | **비밀번호 변경 페이지** | 90일 만료 시 강제 이동되는 페이지. 변경 완료 후 기존 세션 전체 폐기 → 재로그인 안내 |
| `index.html` | **메인 대시보드** | 로그인 후 진입하는 메인 화면. `role`에 따라 표시 메뉴가 달라짐. 환자는 예약 달력/예약 관리, 의료진은 진료 일정/환자 목록 표시 |

### JavaScript / CSS

| 파일 | 역할 | 주요 기능 |
|------|------|-----------|
| `js/config.js` | **API 공통 설정** | `BASE_URL` (백엔드 서버 주소)과 `API_KEY` 상수 정의. **환경 전환 시 이 파일만 수정하면 됨** (로컬 → 운영 서버) |
| `js/script.js` | **메인 페이지 핵심 로직** | ① 로그인 가드 (`requireLogin` — 미인증 시 login.html로 자동 이동) ② 공통 API 호출 함수 (`apiCall` — 401 응답 시 토큰 자동 갱신 후 1회 재시도) ③ 예약 달력 렌더링 ④ 예약 생성·수정·취소 모달 처리 ⑤ 의료진용 환자 목록·상세 조회 ⑥ 로그아웃 처리 (서버 세션 폐기 + 쿠키 삭제) |
| `js/main.js` | **공통 UI 동작** | 헤더 스크롤 애니메이션, 모바일 햄버거 메뉴 개폐, 상단 공지 배너 속도 조정. 로그인 여부와 무관한 UI 전용 |
| `css/style.css` | **공통 스타일시트** | 헤더 스크롤 효과, 달력 셀 디자인, 모달, 모바일 메뉴 등 사이트 전반 CSS |

## 데이터 흐름 요약

```
브라우저 (frontend)              백엔드 (backend)                   DB
────────────────                 ───────────────                    ──
login.html
  │  POST /auth/login
  │ ─────────────────────────→  routers/auth.py
  │                               └─ core/security.py  (JWT 생성)
  │                               └─ models/db.py      (Session 저장)
  │  ←── Set-Cookie ─────────                                  PostgreSQL
  │      httpOnly 쿠키 발급
  │      (access_token, refresh_token)
  │
index.html + script.js
  │  GET /portal/appointments
  │ ─────────────────────────→  routers/portal.py
  │                               └─ core/security.py  (쿠키 검증)
  │                               └─ models/db.py      (예약 조회)
  │  ←── JSON 응답 ────────────
  │
  │  (모든 API 요청마다 자동)   core/middleware.py
  │                               └─ AuditLog: audit_logs 테이블에 기록
  │  ←── 모든 응답 헤더에 ─────  └─ SecurityHeaders: 보안 헤더 자동 추가
```

## 인증 방식 요약 (JWT + httpOnly Cookie + Refresh Token DB 하이브리드)

```
로그인 성공
    │
    ├─ access_token  (JWT, 30분 유효) → httpOnly 쿠키로 저장
    │                                    JS에서 읽기 불가 → XSS 방어
    └─ refresh_token (랜덤, 8시간 유효) → httpOnly 쿠키로 저장
                                          path=/auth/refresh로 범위 제한

API 호출 시
    브라우저가 쿠키 자동 전송 → 서버가 access_token 검증

access_token 만료 시
    POST /auth/refresh → 서버가 refresh_token 검증 (DB 조회)
                       → 구 토큰 폐기 + 새 토큰 발급 (Rotation)

강제 로그아웃 필요 시
    DB의 sessions 테이블에서 is_revoked = true 설정
    → 해당 refresh_token은 영구 무효화
```

| 방식 | 장점 | 이 프로젝트 적용 이유 |
|------|------|----------------------|
| JWT만 사용 | DB 조회 없어 빠름 | 로그아웃·강제 만료 불가 → 의료정보 보안 부적합 |
| DB 세션만 사용 | 즉시 무효화 가능 | 모든 요청마다 DB 조회 → 성능 부하 |
| **JWT + DB 하이브리드** | 성능 + 제어권 모두 확보 | **현재 적용 방식** — ISMS-P 2.5.x 대응 |

## 역할(Role)별 접근 범위

| Role | 설명 | 접근 가능 메뉴 |
|------|------|----------------|
| `patient` | 환자 (일반 회원가입) | 예약 달력, 예약 생성·수정·취소, 내 정보 |
| `doctor` | 의사 (관리자가 계정 생성) | 진료 일정, 담당 환자 목록·상세 조회 |
| `nurse` | 간호사 (관리자가 계정 생성) | 진료 일정, 담당 환자 목록·상세 조회 |
| `admin` | 관리자 | 모든 메뉴 + 계정 생성 (`/admin/users`) |
