## 1. 현재 인프라 구성 현황

이 문서에서 논의하는 모든 결정은 **이미 구축된 인프라를 기반**으로 합니다.

### 1-1. 네트워크 (VPC)

| 항목 | 내용 |
|---|---|
| VPC | `aws-vpc-01` / CIDR `10.0.0.0/16` / 리전 `ap-south-2` |
| Public 서브넷 | `10.0.1.0/24` ~ `10.0.3.0/24` (3 AZ) |
| App 서브넷 (Private) | `10.0.11.0/24` ~ `10.0.13.0/24` (3 AZ) |
| DB 서브넷 (Private) | `10.0.21.0/24` ~ `10.0.23.0/24` (3 AZ) |

3개의 레이어(Public / App / DB)로 서브넷이 명확히 분리되어 있으며, 각 레이어는 3개의 가용 영역(AZ)에 분산되어 있습니다.

### 1-2. 데이터베이스

| 항목 | 내용 |
|---|---|
| 서비스 | Aurora PostgreSQL (`aws-aurora-01`) |
| Writer 인스턴스 | `ap-south-2a` |
| 서브넷 그룹 | DB 서브넷 3 AZ |
| 접근 허용 | App 서브넷, 온프레미스 VPN |
| 삭제 방지 | `deletion_protection = true` |
| 암호화 | Secrets Manager 연동 |
| RDS Proxy | 별도 스크립트로 토글 관리 |
| 백업 | `skip_final_snapshot = false` |

### 1-3. 보안 모니터링 (Wazuh)

| 항목 | 내용 |
|---|---|
| 인스턴스 | `aws-wazuh-02` (t3.xlarge, Ubuntu 22.04) |
| 서브넷 | App 서브넷 (`ap-south-2b`) |
| 구성 | Master-Worker 클러스터 구조 |
| 관리 방식 | Ansible + AWS SSM (SSH 없이 관리) |
| 알림 | Slack Webhook 연동 |

### 1-4. 온프레미스 연결 (VPN)

| 항목 | 내용 |
|---|---|
| 방식 | AWS Site-to-Site VPN (IPSec) |
| Customer Gateway | 온프레미스 공인 IP 등록 |
| 라우팅 | 정적 라우팅 (온프레미스 대역 등록) |
| 적용 범위 | Public / App / DB 모든 라우팅 테이블 |

온프레미스 ↔ AWS 간 VPN이 이미 구축되어 있으며, DB 레이어까지 온프레미스에서 직접 접근 가능합니다. 이는 온프레미스 원본 데이터 → AWS 비식별화 동기화 구조의 핵심입니다.

### 1-5. Bastion Host

개발/운영 시 DB 직접 접근을 위한 Bastion Host가 별도 스크립트로 토글 관리됩니다. 평소엔 꺼두고 필요할 때만 생성하여 보안 노출을 최소화합니다.

## 2. 웹 서비스 배포 방식 결정

### 2-1. 검토한 옵션

| 옵션 | 검토 결과 |
|---|---|
| **EKS** | Kubernetes 전문 인력 필요, 컨트롤 플레인 비용 ~$70/월, 중소형 병원에 과함 |
| **ECS Fargate** | 서버리스라 OS 레벨 접근 불가 → Wazuh 에이전트 설치 불가 |
| **App Runner** | 구성 단순하지만 VPC 내부 RDS 연동이 복잡, Wazuh 설치 불가 |
| **EC2 + Docker Compose** | 오토스케일링 미지원, 자동 복구 직접 구현 필요 |
| **ECS EC2** ✅ | Wazuh 설치 가능, 오토스케일링/자동 복구 기본 제공, 운영 부담 적정 |

### 2-2. ECS EC2를 선택한 이유

#### (1) Wazuh 에이전트 설치 가능

현재 프로젝트는 Wazuh Master-Worker 클러스터가 이미 구축되어 있습니다. 웹 서버에 Wazuh 에이전트를 설치하려면 **OS 레벨 접근이 필수**입니다.

```
Fargate, App Runner → OS 접근 불가 → Wazuh 에이전트 설치 불가 ❌
ECS EC2            → EC2 OS에 직접 설치 가능 ✅
```

#### (2) 오토스케일링 + 자동 복구 기본 제공

의료 서비스 특성상 예약 시스템이 다운되면 병원 운영에 직접 영향이 생깁니다.

| 장애 유형 | ECS EC2 대응 |
|---|---|
| 컨테이너 다운 | ECS가 자동으로 Task 재시작 |
| EC2 하드웨어 장애 | ASG가 새 EC2 자동 투입 |
| 트래픽 급증 | ECS Auto Scaling으로 Task 자동 증설 |

#### (3) EKS 대비 관리 포인트 감소

| 관리 항목 | EKS | ECS EC2 |
|---|---|---|
| 컨트롤 플레인 업그레이드 | 직접 관리 | AWS 완전 관리 |
| 네트워킹 (CNI) | 플러그인 별도 관리 | VPC 자동 처리 |
| 로드밸런서 연동 | ALB Controller 설치 필요 | 기본 연동 |
| 배포 도구 | kubectl, helm, kustomize... | AWS CLI / 콘솔 |
| 학습 곡선 | 높음 | 낮음 |

EKS는 마이크로서비스가 10개 이상이거나 서비스 메시가 필요한 대규모 시스템에 적합합니다. **이 프로젝트는 FastAPI 앱 하나**이므로 ECS EC2로 충분합니다.

## 3. 단일 컨테이너 (모놀리식) 유지 결정

### 3-1. 현재 코드 구조

```
app/portal/backend/
├── main.py               ← FastAPI 앱 1개
├── core/
│   ├── database.py       ← 공통 DB 연결
│   └── security.py       ← 공통 보안 유틸
├── models/
│   └── db.py             ← 공통 DB 모델
└── routers/
    ├── auth.py            ← 인증 (환자 + 의료진 공통)
    ├── portal.py          ← 환자/의료진 포털
    └── admin.py           ← 관리자
```

세 라우터가 **동일한 DB 모델, 동일한 보안 유틸, 동일한 DB 연결을 공유**합니다.

### 3-2. 마이크로서비스로 쪼개면 생기는 문제

```
억지로 분리할 경우:

auth-service    ──┐ DB 모델 복제 필요
portal-service  ──┼ 서비스 간 네트워크 호출 추가
admin-service   ──┘ 장애 포인트 3배 증가
                   운영 비용 3배 증가
```

### 3-3. 마이크로서비스가 필요한 조건 (현재 프로젝트와 비교)

| 마이크로서비스 필요 조건 | 이 프로젝트 |
|---|---|
| 서비스별 독립 배포 필요 | ❌ 동시 배포로 충분 |
| 서비스별 트래픽 차이 커서 독립 스케일링 필요 | ❌ auth/portal/admin 트래픽 패턴 유사 |
| 서비스별 다른 기술 스택 | ❌ 전부 Python/FastAPI |
| 서비스별 독립된 대규모 개발팀 | ❌ 소규모 팀 |
| 수백만 사용자 대규모 트래픽 | ❌ 중소형 병원 규모 |

> **결론: 마이크로서비스는 복잡한 문제를 해결하는 도구입니다. 지금은 그 복잡한 문제가 없습니다.**

## 4. 고가용성 (HA) 구성

### 4-1. 컨테이너 1개 = 인스턴스 1개가 아님

"단일 컨테이너"는 **이미지(앱) 종류가 1가지**라는 의미이지, 실행 개수가 1개라는 뜻이 아닙니다.

```
FastAPI 이미지 1종류
  ├── Task (ap-south-2a)  ← 실행 중인 컨테이너
  ├── Task (ap-south-2b)  ← 실행 중인 컨테이너
  └── Task (ap-south-2c)  ← 실행 중인 컨테이너
```

### 4-2. AZ 장애 시나리오

```
ap-south-2b AZ 장애 발생
  ├── ALB: ap-south-2b 트래픽을 2a, 2c로 즉시 전환
  ├── ECS: ap-south-2b Task 종료 감지 → 다른 AZ에 자동 재배치
  └── Aurora: 자동 페일오버 (30초 이내)
        └── 사용자는 장애를 느끼지 못함
```

### 4-3. HA 보장 조건

| 항목 | 설정값 |
|---|---|
| 최소 실행 Task 수 | 3개 (AZ당 1개) |
| ALB 헬스체크 엔드포인트 | `/health` (이미 구현됨) |
| ECS 최소 가용 비율 | 50% 이상 유지 |
| ASG | EC2 장애 시 자동 교체 |
| Aurora | 3 AZ 클러스터 (이미 구성됨) |

현재 Aurora 클러스터가 이미 3 AZ에 구성되어 있으므로, ECS EC2도 동일하게 3 AZ에 배포하면 전체 스택의 HA가 완성됩니다.

## 5. 환자 / 의료진 접근 분리

### 5-1. 결정 사항

의료진은 **병원 내부망에서만** 접근합니다. 이미 구축된 온프레미스 ↔ AWS VPN을 활용합니다.

### 5-2. 구현 방식

코드를 분리하지 않고 **ALB 경로 기반 규칙**으로 네트워크 레벨에서 분리합니다.

```
인터넷 (모든 IP)
  └── ALB
        ├── /auth/register        → 허용 (환자 회원가입)
        ├── /auth/login           → 허용 (공통 로그인)
        ├── /portal/appointments  → 허용 (환자 예약)
        ├── /portal/doctor/*      → 병원 VPN IP만 허용 ← 차단
        └── /admin/*              → 병원 VPN IP만 허용 ← 차단
```

```
병원 내부망 → VPN → AWS
  └── ALB → /portal/doctor/*, /admin/* → 접근 허용
```

### 5-3. 역할 기반 접근 제어 (이미 구현됨)

네트워크 레벨 분리 외에도, 코드 레벨에서 이중으로 접근이 제어됩니다.

```python
# portal.py — 의료진 전용 엔드포인트
if current_user.get("role") not in ("doctor", "nurse", "admin"):
    raise HTTPException(status_code=403, detail="권한이 없습니다.")
```

```javascript
// script.js — 프론트엔드 role별 메뉴 표시
if (role === 'patient') {
    navPatientMenu?.classList.remove('hidden');
} else if (role === 'doctor' || role === 'nurse') {
    navMedicalStaffMenu?.classList.remove('hidden');
}
```

### 5-4. ISMS-P 관점 이점

```
✅ 내부 시스템과 외부 시스템 네트워크 레벨 분리
✅ 의료진의 환자 데이터 접근이 내부망으로 제한
✅ 외부 공격 노출 면적 감소
✅ 네트워크 / 코드 이중 접근 제어
```

## 6. HTTPS 적용 방식

### 6-1. nginx가 필요 없는 이유

ALB가 TLS를 직접 처리(Terminate)하므로 nginx reverse proxy가 불필요합니다.

```
클라이언트 ──HTTPS(443)──▶ ALB (ACM 인증서로 TLS 종료)
                                    │
                               HTTP(8000)
                                    │
                           FastAPI 컨테이너 (Uvicorn)
```

nginx를 추가하면 관리 포인트만 늘어나고 실질적인 이점이 없습니다.

## 7. DR (재해복구) — GCP PostgreSQL 논리 복제

### 7-1. 구성

```
AWS Aurora PostgreSQL (Primary)
  └── PostgreSQL 논리 복제 (WAL)
        └── GCP Cloud SQL PostgreSQL (DR Replica)
```

### 7-2. 선택 이유

| 방식 | RPO | 복잡도 | 선택 |
|---|---|---|---|
| pg_dump 주기 백업 | 수 시간 | 낮음 | ❌ |
| AWS DMS | 수 분 | 중간 | ❌ |
| PostgreSQL 논리 복제 | **수초~수 분** | 중간 | ✅ |

의료 데이터 특성상 RPO는 최소화해야 합니다. PostgreSQL 네이티브 논리 복제는 별도 툴 없이 WAL 기반으로 실시간에 가까운 복제가 가능합니다.

### 7-3. 네트워크

AWS ↔ GCP VPN 터널을 통해 안전하게 복제 트래픽을 전송합니다. 이미 온프레미스 ↔ AWS VPN 구성 경험이 있으므로 동일한 패턴으로 구성 가능합니다.

## 8. 최종 아키텍처 요약

```
┌─────────────────────────────────────────────────────────────┐
│                        인터넷                                │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTPS
                    ┌──────▼──────┐
                    │  ALB + WAF  │  (ACM 인증서, 경로 기반 접근 제어)
                    └──────┬──────┘
          ┌────────────────┼────────────────┐
          │                │                │
    ┌─────▼─────┐    ┌─────▼─────┐    ┌─────▼─────┐
    │ ECS Task  │    │ ECS Task  │    │ ECS Task  │
    │ (2a)      │    │ (2b)      │    │ (2c)      │
    │ FastAPI   │    │ FastAPI   │    │ FastAPI   │
    └─────┬─────┘    └─────┬─────┘    └─────┬─────┘
          │  EC2 (Wazuh Agent 설치)          │
    ┌─────▼──────────────────────────────────▼─────┐
    │         Aurora PostgreSQL 클러스터             │
    │         (Writer: 2a / Replica: 2b, 2c)        │
    └──────────────────────┬────────────────────────┘
                           │ 논리 복제
                    ┌──────▼──────┐
                    │ GCP Cloud   │
                    │ SQL (DR)    │
                    └─────────────┘

[온프레미스] ──VPN──▶ [AWS VPN Gateway] ──▶ 내부망 전용 경로
[의료진 PC]  ──────────────────────────────▶ /portal/doctor/*, /admin/*

[프론트엔드] S3 + CloudFront (정적 HTML/CSS/JS)
[보안 모니터링] Wazuh Master-Worker 클러스터 (기존 구성 활용)
```

## 9. 향후 확장 계획

현재 구조는 모놀리식이지만, 아래 조건이 충족되면 서비스 분리를 검토합니다.

| 조건 | 분리 대상 |
|---|---|
| 예약 트래픽이 폭증하여 독립 스케일링 필요 | portal 서비스 분리 |
| 관리자 기능이 대폭 확장 | admin 서비스 분리 |
| 병원이 다지점으로 확장 | 멀티테넌트 또는 서비스 분리 검토 |

> **지금 당장 쪼개는 것은 오버엔지니어링입니다. 필요할 때 분리하는 것이 올바른 접근입니다.**

## 10. 결론

| 항목 | 결정 | 이유 |
|---|---|---|
| 컨테이너 오케스트레이션 | **ECS EC2** | Wazuh 에이전트 + HA + 적정 운영 복잡도 |
| 서비스 구조 | **모놀리식 (단일 컨테이너)** | 공유 의존성, 소규모 팀, 현 규모에 적합 |
| HTTPS | **ALB + ACM** | nginx 불필요, ALB가 TLS 처리 |
| 프론트엔드 | **S3 + CloudFront** | 정적 파일, 서버 불필요 |
| 접근 분리 | **ALB 경로 규칙 + VPN** | 코드 분리 없이 네트워크 레벨 분리 |
| DR | **GCP 논리 복제** | RPO 수초~수 분 |
| 가용성 | **3 AZ** | 의료 서비스 특성상 단일 AZ 장애 허용 불가 |
